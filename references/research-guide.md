# 深度研究行为指南

推理驱动的深度研究工作流。LLM 先推理、联网扩展，vault 作为补充验证。产出叙事驱动的深度分析报告。

## 定位

| 场景 | 你做什么 | 脚本做什么 |
|------|---------|-----------|
| 深度研究 | 推理形成假说、联网搜索、纵向/横向分析、撰写报告 | 研究项目初始化、vault 补充验证、报告生成、PDF 导出 |
| 查询 | 搜索 vault + 综合回答 | → 加载 `query-guide.md` |
| 入库 | 抓取 + 编译 + 回写 | → 加载 `ingest-guide.md` |

---

## 核心原则

**推理驱动，非 vault 驱动**：深度研究的前提是现有知识不够。你应从用户问题出发推理，联网获取信息构建分析框架。vault 仅在联网研究之后作为补充验证。禁止用 vault 已有内容锚定分析方向。

**横纵双轴**：分析沿两条轴展开——纵向追时间深度（起源→演进→决策逻辑→阶段划分），横向追同期广度（竞品对比→生态位→趋势判断）。不做 6 个平行维度的清单式搜索。

**叙事驱动**：报告是有因果脉络的深度分析，不是按维度分组的信息汇编。纵向部分要有故事弧线，有起承转合。

## 触发条件

### 用户主动触发

触发词：`深入研究`、`深度分析`、`deep research`、`系统分析`、`我需要做关于 X 的决策`

### 对话中自动触发

查询过程中，按 `references/query-guide.md` Step 6 的信号规则自动判断。

| 信号 | 判断规则 | 优先级 |
|------|---------|--------|
| 外部事实依赖 | 答案需要验证外部事实，vault 中没有 | high |
| 多源矛盾 | ≥ 2 个来源对同一问题有矛盾回答 | high |
| 低覆盖度 | vault 中 < 2 个相关页面 | medium |
| 用户追问 | 同一话题追问 ≥ 3 轮 | medium |

触发规则：任一 high 命中 / ≥ 2 个 medium。

## 执行流程

详细协议见 `references/deep-research-protocol.md`。以下是行为概要：

### Phase 0: 激活与对象识别

快速了解 vault 已有覆盖（非分析基础），判断研究对象类型。

```
python scripts/wiki_query.py "<topic>" --mode briefing --vault <vault>
python scripts/deep_research.py list --vault <vault> --json
```

读 `wiki/hot.md` + `wiki/index.md` 了解已有覆盖。

**对象类型判断**：产品、公司、概念、人物——决定后续分析侧重点。

### Phase 1: 意图扩展

问用户 1-2 个高价值问题：
- 为什么这个问题重要？
- 好的答案长什么样？
- 真正关心的是什么？

确认研究对象和类型（如用户输入不够明确）。记录用户内在模型到研究上下文。

### Phase 2: 假说形成

按 `references/prompts/research_hypothesis.md` 约束形成假说。

1. 从用户问题出发推理形成，**禁止从 vault 已有内容归纳**
2. 形成 1-2 个纵深可证伪假说（宁可深挖一个到不可再分）
3. 每个假说必须通过对立性检验 + 确认查询和反驳查询

```
python scripts/deep_research.py init --vault <vault> --topic "<topic>" --object-type "<type>" --hypotheses-json hypotheses.json
```

### Phase 3: 联网信息收集（横纵双线并行）

**这是研究的信息基础。必须联网搜索，不能仅靠已有知识。**

搜索分两条线：
- **纵向线**：起源、发展历程、关键事件、决策逻辑（3-5 个查询）
- **横向线**：竞品识别、对比评测、用户口碑（3-5 个查询）
- **补充线**（视需要）：学术原理、最新动态、批评声音（2-3 个查询）

执行流程：
1. 批量 WebSearch，每条线覆盖不同关键词组合
2. 筛选高价值结果（优先一手来源）
3. WebFetch 提取内容，**不经过入库管线**，直接用于分析
4. 在研究上下文中记录发现的来源和内容摘要

信息充分性自检：
- 纵向能讲出完整故事？横向竞品列表完整？关键事实有可靠来源？
- 不够 → 补充搜索，最大 3 轮

### Phase 4: 纵向分析（Diachronic）

沿时间轴，完整还原研究对象从诞生到现在的发展全貌。报告主体部分。

内容要求：
- **起源追溯**：诞生背景、创始团队、行业环境
- **演进历程**：按时间顺序梳理所有关键节点
- **决策逻辑**：每个关键节点为什么这么选，约束条件是什么，哪些早期决策锁定了后来的方向
- **阶段划分**：分为几个阶段，每阶段有核心特征和核心矛盾

关键断言带证据标签：`[Fact]` / `[Inference]` / `[Assumption]` / `[Disputed]` / `[Gap]`

### Phase 5: 横向分析（Synchronic）

以当前时间点为切面，与竞品/同类全面对比。

先判断竞品场景：
- **A：无直接竞品** → 分析为什么没有，未来可能从哪个方向冒出
- **B：少量竞品（1-2 个）** → 逐一深入对比
- **C：竞品充分（3+）** → 选最具代表性的 3-5 个对比

对比维度：核心差异、用户视角、生态位分析、趋势判断。

### Phase 6: 横纵交汇洞察 + 剧本推演

报告精华段。把纵向脉络和横向格局结合，产出新判断。

核心问题：
1. 历史如何塑造了当下的竞争位置？
2. 竞品的纵向对比——不同起源路径如何导致今天各自的特点？
3. 优势/劣势的历史根源

三剧本推演（从横纵交叉中产出）：
- **最可能**：延续当前轨迹，3-6 个月后的具体状态
- **最危险**：触发条件 + 失败机制 + 量化扰动
- **最乐观**：需要哪些条件同时成立，区分可控/不可控

边界条件：`"当 [具体可观察条件] 时，结论 [成立/失效]"`

```
python scripts/deep_research.py record-scenarios --vault <vault> --topic "<topic>" --scenarios-json scenarios.json
```

### Phase 7: Vault 补充验证

**在联网研究之后**做 vault 补充，避免 vault 已有内容锚定分析方向。

```
python scripts/deep_research.py collect-vault-evidence --vault <vault> --topic "<topic>" --claims "<claim1>" "<claim2>"
```

交叉验证：
- 联网发现与 vault 已有知识是否一致？不一致标注 `[Disputed]`
- vault 中是否有联网搜索未覆盖的独特视角？
- 哪些已有判断需要因新证据而修正？

知识缺口识别：识别"vault 里缺什么使我无法判断假说的真假"，而非"vault 里有什么能支持我的假说"。

### Phase 8: 收敛与报告打包

1. 证据密度检查
2. 打包报告（结构见下方）
3. 所有强断言带证据标签
4. 执行 5 项质量门控
5. 生成 PDF

```
python scripts/deep_research.py finalize-report --vault <vault> --topic "<topic>" --report-file report.md --run-closure
```

## 报告结构

```
# {研究对象} — 深度研究报告

## 执行摘要（≤300 字）
[核心判断 + 推荐行动 + 最大风险]

## 一、一句话定义
[用一句话说清楚这个东西是什么]

## 二、纵向分析：从诞生到当下
[完整的纵向叙事——起源、演进、决策逻辑、阶段划分]
[6000-15000 字，视对象复杂度调整]

## 三、横向分析：竞争图谱
[竞品场景判断 + 核心差异对比 + 用户视角 + 生态位 + 趋势判断]
[3000-10000 字，视竞品数量调整]

## 四、横纵交汇洞察
[历史如何塑造当下竞争位置 + 竞品纵向对比 + 优势/劣势历史根源]
[三剧本推演 + 边界条件声明]
[1500-3000 字]

## 五、实施路径（如适用）
[阶段分解 + 路径依赖风险 + 应急分支 + 假设声明]
[研究概念/人物时此章可省略]

## 六、信息来源
[所有引用的来源清单，标注 URL 和访问时间]

## 附录：质量门控（5 项检验）
```

**结构适配**：
- 产品/公司：完整结构
- 概念：省略或简化"实施路径"，扩展"交汇洞察"
- 人物：省略"实施路径"，横向分析改为"与同领域人物对比"

---

## 质量门控（5 项检验）

1. **叙事完整性** — 纵向分析有完整故事弧线？无重要节点被跳过？
2. **决策逻辑检验** — 每个关键节点还原了"为什么这么选"，不只是"发生了什么"？
3. **反面证据检验** — 无故意弱化的反面论据？横向分析公正呈现竞品优势？
4. **证据标签检验** — 读者能一眼区分 [Fact] 和 [Assumption]？
5. **边界条件检验** — 强判断标注了失效条件？来自三剧本推演？

---

## 证据标签

所有强断言必须携带标签：
- `[Fact]` — Tier 1/2 来源的直接事实
- `[Inference]` — 从多个事实推理
- `[Assumption]` — 未经验证
- `[Disputed]` — 可信来源有争议
- `[Gap]` — 缺乏证据

## 边界

- **你负责**：假说形成、联网搜索策略、纵向/横向分析、报告撰写、质量门控
- **脚本负责**：研究项目初始化、vault 补充验证、报告打包、PDF 生成
- **用户在 loop 中**：Phase 1 意图扩展、Phase 8 报告审阅——不要跳过用户确认
- **入库在研究后**：联网研究中获取的高价值来源，在报告完成后批量入库（非研究过程中逐条入库）
