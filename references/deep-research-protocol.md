# Deep Research Protocol

推理驱动的深度调研协议。LLM 先推理、联网扩展，vault 作为补充验证数据源。产出叙事驱动的深度分析报告。

## 核心原则

**推理驱动，非 vault 驱动**：深度研究的前提是现有知识不够。LLM 应从用户问题出发推理，联网获取信息构建分析框架，vault 仅在联网研究之后作为补充验证。禁止用 vault 已有内容锚定分析方向。

**横纵双轴，非平行维度**：分析沿两条轴展开——纵向追时间深度（起源→演进→决策逻辑→阶段划分），横向追同期广度（竞品对比→生态位→趋势判断）。两轴交叉产出原创洞察。不做 6 个平行维度的清单式搜索。

**叙事驱动，非流程驱动**：报告是有因果脉络的深度分析，不是按维度分组的信息汇编。纵向部分要有故事弧线，有起承转合。

## 触发条件

触发词：`"深入研究 X"` / `"深度分析 X"` / `"deep research X"` / `"系统分析 X"` / `"我需要做关于 X 的决策/判断/方案"`

自动触发信号（查询过程中）：

| 信号 | 判断规则 | 优先级 |
|------|---------|--------|
| 外部事实依赖 | 答案需要验证外部事实，vault 中没有 | high |
| 多源矛盾 | ≥ 2 个来源对同一问题有矛盾回答 | high |
| 低覆盖度 | vault 中 < 2 个相关页面 | medium |
| 用户追问 | 同一话题追问 ≥ 3 轮 | medium |

触发规则：任一 high 命中 / ≥ 2 个 medium。

与现有模式区分：
- 简单事实查询 → `brief`
- 结构化概览 → `briefing`
- 多视角聚合（vault 内） → `digest`
- 广泛探索（无具体命题） → `autoresearch`
- 推理驱动+联网扩展+横纵分析+证据标注 → **deep-research**

## 8 阶段工作流

### Phase 0: 激活与对象识别

**目的**：快速了解 vault 已有覆盖（非分析基础），判断研究对象类型。

脚本调用：
```
python scripts/wiki_query.py "<topic>" --mode briefing --vault <vault>
python scripts/deep_research.py list --vault <vault> --json
```
读取 `wiki/hot.md` + `wiki/index.md`。

**对象类型判断**（决定后续分析侧重点）：

| 类型 | 纵轴侧重 | 横轴侧重 |
|------|---------|---------|
| 产品 | 版本迭代、技术路线演变、用户增长、关键产品决策 | 功能对比、性能、用户体验、定价 |
| 公司 | 创始团队、融资历程、战略转向、组织变革、关键人事 | 商业模式差异、市场份额、营收、组织架构 |
| 概念 | 概念起源（谁提出、基于什么理论）、流行过程、争论与演变 | 与相近概念区别、适用场景、不同阵营论证 |
| 人物 | 个人经历、职业轨迹、关键决策、成长曲线、公开言论变化 | 与同领域人物对比（做事方式、风格、成就、路线差异） |

### Phase 1: 意图扩展

LLM 推理（无脚本调用）：
- 问用户 1-2 个高价值问题：为什么重要、真正关心什么、好的答案什么样
- 确认研究对象和类型（如用户输入不够明确）
- 记录用户内在模型到研究上下文页

### Phase 2: 假说形成

按 `references/prompts/research_hypothesis.md` 约束形成假说。

**关键约束**：
- 从用户问题出发推理形成，禁止从 vault 已有内容归纳
- 1-2 个纵深假说，宁可深挖一个到不可再分
- 每个假说必须通过对立性检验
- 每个假说至少 1 个确认查询 + 1 个反驳查询

**假说卡格式**：
```json
{
  "claim": "具体的可证伪断言",
  "type": "causal | structural | comparative | threshold",
  "confidence": 25,
  "confirm_evidence": "什么证据能确认",
  "contradict_evidence": "什么证据能反驳",
  "confirm_queries": ["确认查询 1"],
  "contradict_queries": ["反驳查询 1"]
}
```

脚本调用：
```
python scripts/deep_research.py init --vault <vault> --topic "<topic>" --object-type "<type>" --hypotheses-json hypotheses.json
```

### Phase 3: 联网信息收集（横纵双线并行）

**核心**：这是研究的信息基础。必须联网搜索，不能仅靠已有知识。

**并行搜索策略**（按对象类型调整查询）：

**纵向线**（时间轴信息）：
- 起源、创始人/发起者背景、发展历程
- 关键事件、版本迭代、融资、战略转向、危机
- 决策逻辑：每个关键节点为什么这么选
- 查询示例：`"{对象} 起源 历史"`、`"{对象} 发展历程 关键事件"`、`"{对象} 创始人 背景"`

**横向线**（竞争对比信息）：
- 竞品识别、各竞品特点和用户口碑
- 行业对比评测、市场份额、商业模式差异
- 查询示例：`"{对象} vs 竞品"`、`"{领域} 竞争格局 2026"`、`"{对象} 用户评价"`

**补充线**（视需要）：
- 学术/技术原理（技术话题必查 arxiv）
- 最新动态、政策变化（时间敏感话题）
- 批评声音、失败案例、局限性（任何研究都不能缺）
- 查询示例：`"{对象} 批评 问题"`、`"{技术} 原理 论文"`、`"{对象} 最新消息 2026"`

**搜索执行流程**：

1. **批量 WebSearch**：每条线 3-5 个查询，覆盖不同关键词组合
2. **筛选高价值结果**：每个查询选 1-2 个最相关 URL（优先一手来源）
3. **WebFetch 提取内容**：直接从页面提取关键信息，不经过入库管线
4. **记录发现**：在研究上下文中记录每条发现的来源和内容摘要

**信息来源优先级**：

| 信息类型 | 一手来源 |
|---------|---------|
| 产品更新/技术决策 | 官方博客、GitHub Release Notes、创始人推文 |
| 融资/商业数据 | 公司官方公告、SEC/工商文件 |
| 用户口碑 | GitHub Issues、Reddit、Twitter/X、知乎 |
| 行业分析 | 权威媒体原创报道（非转载） |
| 学术/技术 | arXiv 论文、Google Scholar、学术会议论文集 |

**信息充分性自检**（搜索完成后）：
- 纵向：能讲出一个完整的故事吗？有没有明显的信息断层？
- 横向：竞品列表完整吗？每个竞品的信息够做对比吗？
- 来源：关键事实有可靠来源支撑吗？有没有只靠单一来源就下判断的？
- 信息不够 → 补充搜索，最大 3 轮

### Phase 4: 纵向分析（Diachronic）

沿时间轴，完整还原研究对象从诞生到现在的发展全貌。这是报告的主体部分。

**内容要求**：

**起源追溯**：诞生的背景是什么？基于什么技术/理念/需求而来？创始团队或核心推动者是谁？当时的行业环境是什么？

**诞生节点**：明确的首次发布/成立/提出时间，最初的形态和定位。

**演进历程**：从诞生到现在，按时间顺序梳理所有关键节点——重大版本更新、融资事件、团队变动、战略转型、技术架构变化、用户规模里程碑、重大合作或收购、公关危机或争议。

**决策逻辑**：在每个关键节点上，还原决策背后的原因。为什么选了A而不是B？当时面对的约束条件是什么？哪些早期决策"锁定"了后来的发展方向？

**阶段划分**：把整个历程分为几个阶段（萌芽期、快速增长期、转型期等），每个阶段有核心特征和核心矛盾。

**证据标注**：关键断言带 `[Fact]` / `[Inference]` / `[Assumption]` / `[Disputed]` / `[Gap]` 标签。

**篇幅**：视对象复杂度调整。历史越长、节点越多的对象写得越长。核心原则是把故事讲完整、讲透。

### Phase 5: 横向分析（Synchronic）

以当前时间点为切面，将研究对象与竞品/同类进行全面对比。

**竞品场景判断**：

- **场景 A：无直接竞品** → 分析为什么没有竞品（品类太新、壁垒太高、市场太小？），未来最可能从哪个方向冒出竞争者，有没有间接替代方案。
- **场景 B：少量竞品（1-2 个）** → 逐一深入对比。
- **场景 C：竞品充分（3+）** → 选取最具代表性的 3-5 个进行对比，其余简要提及。

**对比维度**（至少覆盖）：
- 核心差异：技术路线/核心方法论/底层逻辑、产品形态/商业模式、目标用户/适用场景
- 用户视角：真实用户口碑、社区评价、使用体验中被提及最多的优点和槽点
- 生态位分析：在赛道版图中的位置，当前格局（百花齐放/两强争霸/一家独大）
- 趋势判断：基于横向对比的机会和风险

**证据标注**：同 Phase 4。

### Phase 6: 横纵交汇洞察 + 剧本推演

报告的精华段。把纵向发展脉络和横向竞争格局结合起来，产出新的判断。不是前面内容的缩写版。

**核心问题**：

1. **历史如何塑造了当下的竞争位置**：纵向历程中的哪些决策和事件，决定了它今天在横向对比中的位置？
2. **竞品的纵向对比**：主要竞品的起源和演变路径有什么不同？这些不同如何导致了今天各自的特点？
3. **优势的历史根源**：今天的每个核心优势，能追溯到历史上的哪个节点或决策？
4. **劣势的历史根源**：今天的每个核心劣势，能追溯到哪个历史决策？当初的"好决策"有没有变成今天的包袱？

**三剧本推演**（从横纵交叉中产出，不是独立的清单）：

1. **最可能剧本**（基准情景）：延续当前轨迹，3-6 个月后最可能的状态。必须基于 Phase 4-5 的具体数据和趋势推演。
2. **最危险剧本**（压力情景）：什么条件下会失败？失败的具体机制是什么？必须包含至少一个量化扰动。
3. **最乐观剧本**（突破情景）：什么条件下会大幅超越预期？需要哪些条件同时成立？区分可控因素和不可控因素。

**边界条件**：从三个剧本中提取，格式：`"当 [具体可观察条件] 时，结论 [成立/失效/部分成立]"`。必须是可验证的，不能是"如果情况变化"。

脚本调用：
```
python scripts/deep_research.py record-scenarios --vault <vault> --topic "<topic>" --scenarios-json scenarios.json
```

### Phase 7: Vault 补充验证

**目的**：将联网研究的发现与 vault 已有知识交叉验证，补充本地独有的视角和数据。

**为什么在联网研究之后**：如果在联网研究之前做 vault 证据收集，LLM 会被 vault 已有内容锚定分析方向。联网研究建立了独立的分析框架后，vault 才能作为补充而非主导。

脚本调用：
```
python scripts/deep_research.py collect-vault-evidence --vault <vault> --topic "<topic>" --claims "<claim1>" "<claim2>"
```

**交叉验证**：
- 联网发现与 vault 已有知识是否一致？不一致时标注 `[Disputed]`
- vault 中是否有联网搜索未覆盖的独特视角？
- vault 中的哪些已有判断需要因新证据而修正？

**知识缺口识别**：vault 证据收集后，识别"vault 里缺什么使我无法判断假说的真假"，而非"vault 里有什么能支持我的假说"。

### Phase 8: 收敛与报告打包

1. 证据密度检查
2. 打包报告（结构见下方）
3. 所有强断言带证据标签
4. 稳定结论 vs 工作假说显式标注

**质量门控**（5 项检验，替代原有 7 项）：
1. **叙事完整性** — 纵向分析是否有完整的故事弧线？有无重要节点被跳过？
2. **决策逻辑检验** — 每个关键节点是否还原了"为什么这么选"，而不只是"发生了什么"？
3. **反面证据检验** — 有无故意弱化或省略的反面论据？横向分析是否公正呈现了竞品优势？
4. **证据标签检验** — 读者能否一眼区分 [Fact] 和 [Assumption]？
5. **边界条件检验** — 强判断是否标注了失效条件？边界条件是否来自三剧本推演而非泛泛描述？

脚本调用：
```
python scripts/deep_research.py finalize-report --vault <vault> --topic "<topic>" --report-file report.md --run-closure
```

**PDF 生成**：报告写入后自动生成 PDF。路径：
- Markdown: `wiki/research/{slug}--report.md`
- PDF: `wiki/research/{slug}--report.pdf`

## 报告结构

报告采用"深度研究叙事"结构：先给一句话定义，再沿纵轴展开历史叙事，横轴做竞争对比，最后交叉产出洞察。

```
# {研究对象} — 深度研究报告

## 执行摘要（≤300 字）
[核心判断 + 推荐行动 + 最大风险]

## 一、一句话定义
[用一句话说清楚这个东西是什么]

## 二、纵向分析：从诞生到当下
[完整的纵向叙事——起源、演进、决策逻辑、阶段划分]
[证据标注：[Fact] / [Inference] / [Assumption] / [Disputed] / [Gap]]

## 三、横向分析：竞争图谱
[竞品场景判断 + 核心差异对比 + 用户视角 + 生态位 + 趋势判断]
[证据标注]

## 四、横纵交汇洞察
[历史如何塑造当下竞争位置 + 竞品纵向对比 + 优势/劣势的历史根源]
[三剧本推演（最可能/最危险/最乐观）]
[边界条件声明]

## 五、实施路径（如适用）
[阶段分解 + 路径依赖风险 + 应急分支 + 假设声明]
[注：研究概念/人物时此章可省略或简化]

## 六、信息来源
[所有引用的来源清单，标注 URL 和访问时间]

## 附录：质量门控（5 项检验）
[由 finalize_report 自动生成]
```

**结构适配规则**：
- 研究产品/公司：完整结构，"实施路径"章节保留
- 研究概念：省略或大幅简化"实施路径"，扩展"横纵交汇洞察"
- 研究人物：省略"实施路径"，横向分析改为"与同领域人物对比"

## 证据标签系统

所有强断言必须携带证据标签：
- `[Fact]` — 来自 Tier 1/2 来源的直接事实
- `[Inference]` — 从多个事实推理得出的结论
- `[Assumption]` — 未经验证的假设
- `[Disputed]` — 可信来源之间存在争议
- `[Gap]` — 缺乏证据，需要进一步研究

---

## 来源评级

| Tier | 来源类型 |
|------|---------|
| 1 | 一手研究、分析师报告、标准文档、监管文件、官方博客 |
| 2 | 可靠新闻、从业者博客、会议论文 |
| 3 | 论坛、匿名帖、聚合摘要——仅用于三角验证，不可锚定 Fact |

---

## Challenger Mode

当以下信号出现时，自动进入 Challenger Mode：
- 假设被当作事实使用
- 置信度在没有新证据的情况下膨胀
- 收敛基于稀薄证据
- 忽略反面证据

Challenger Mode 行为：
- 对每个强断言追问证据来源
- 主动寻找反驳论据
- 明确标注哪些是 Assumption 而非 Fact

## 研究完成后入库

联网研究过程中获取的高价值信息，在报告完成后批量入库（非研究过程中逐条入库）：

```
# 对研究中发现的高价值来源逐条入库
python scripts/wiki_ingest.py --url <url> --vault <vault>
python scripts/llm_compile_ingest.py --prepare-only --lean --vault <vault> --raw <raw_path>
python scripts/apply_compiled_brief_source.py --vault <vault> --compiled-json <json_path>
```

入库时在 source 页标记来源为本次深度研究报告，建立与报告页的关联。
